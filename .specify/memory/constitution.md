# 建筑施工质检情报员 宪法

## 核心原则

### I. MVP 优先交付

这是一个演示/MVP 实现。每个决策都必须优先保证核心用户流程的端到端可用性，而非完整性。

- 先跑通主流程；除非阻塞核心路径，否则不处理边界情况
- 不写测试，除非明确要求——类型检查和 lint 通过即可
- 不为可扩展性或未来需求做设计；能快速交付的硬编码可以接受
- 优先"能用"而非"设计正确"；只有当当前方案阻碍进度时才重构

**原因：** 项目明确为复杂项目的 MVP 演示。过度工程化会延误交付且不增加用户价值。

**如何应用：** 面对设计决策时，问："这有助于交付核心路径吗？"如果不是，跳过。

### II. 核心路径正确性

应用必须在以下主要用户流程上行为正确：质检员创建工单 → 施工方处理 → 管理员监控 → 知识沉淀。任何偏离规格的地方都是 bug。

- 工单状态机转换必须严格按照《工单状态机》文档实现
- Agent 对话流程必须遵循 Agent模块/整体设计.md 的架构
- 数据模型必须与《数据定义》逐字段对应
- UI 必须渲染 PRD 页面中描述的所有字段和交互

**原因：** 演示若在核心流程上偏离规格就失去了意义。

**如何应用：** 实现功能时，追溯到具体的 PRD 需求。PRD 说了就做，PRD 没说就不加。

### III. 最小抽象

不做推测性抽象。代码必须以当前功能实际需要的复杂度来编写。

- 不引入没有具体当前用途的模式、包装器或层级
- 不做防御性编程（不可能状态的空值检查、可信内部数据的校验、不会失败操作的 try-catch）
- 不创建只包装单次调用的自定义 hook 或工具函数；直接调用
- 当一个 import 只被使用一次时，优先就近放置代码而非抽取
- 少于 3 个值的常量直接内联，不放配置文件

**原因：** 用户明确要求避免过度抽象和防御性编程。过早抽象让 MVP 代码更难导航。

**如何应用：** 当你准备创建一个可复用辅助函数时，问："它现在是否在多处使用？"如果不是，内联它。

### IV. 实用技术栈

使用 PRD 中指定的技术栈，不替换或添加替代方案。

- **框架：** Next.js（App Router）统一前后端，单一代码库
- **数据库与鉴权：** Supabase（Postgres + Auth + RLS）
- **AI Agent：** LangChain `createAgent`，LangSmith 监控
- **知识 Agent：** 扣子（Coze）平台外部 API
- **AI 总结：** Kimi API（单轮调用，后端 fetch）
- **流式响应：** LangChain `useStream` 处理 SSE
- **UI：** Tailwind + shadcn/ui + lucide-react
- **部署：** Vercel
- **语言：** 全栈 TypeScript；为 MVP 速度可接受 `any`

**原因：** PRD 明确定义了这些选型。技术栈的一致性降低认知负担和集成风险。

**如何应用：** 添加依赖时，先检查 PRD 技术栈是否已提供。如有则使用；如无则添加最精简的包。

## 开发工作流

1. **垂直切片实现：** 一次一个用户故事，全栈贯通。不先做所有模型再做所有 API 再做所有 UI。
3. **频繁提交：** 每个逻辑变更一次提交。MVP 阶段不需要正式 PR 流程。
5. **中文界面：** 所有面向用户的文本必须使用中文。代码注释和变量名使用英文。

## 治理

本宪法管理 MVP 阶段的所有开发决策。

- **修订流程：** 任何团队成员可通过更新本文件并提出明确理由来提议修订。修订在提交后立即生效。
- **版本规则：** 遵循语义化版本——MAJOR 用于原则删除或重新定义，MINOR 用于新增原则或扩展指导，PATCH 用于澄清和措辞修正。
- **合规检查：** 每个用户故事标记完成前，验证其通过类型检查（`tsc --noEmit`）和 lint。MVP 阶段无需正式评审关卡。
- **运行时指导：** 使用 `.claude/CLAUDE.md` 存放 AI 辅助开发指令，作为本宪法的补充（而非覆盖）。
