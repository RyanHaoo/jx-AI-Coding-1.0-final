# Feature Specification: Coze Subagent Tool Integration

**Feature Branch**: `006-coze-subagent`  
**Created**: 2026-04-18  
**Status**: Draft  
**Input**: User description: "把这个 subagent 包装成主 agent 的一个工具，不要把 subagent 的中间消息和状态更提交给主 agent 和前端"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Get Professional Knowledge Answer (Priority: P1)

As a site quality inspector or constructor, I ask a construction quality standard question in the assistant and receive a professional answer from the external knowledge subagent, instead of a mock placeholder.

**Why this priority**: This is the core user value of the feature. Without this, the integration does not improve real-world usefulness.

**Independent Test**: Sign in as any mobile assistant user, ask a standards-related question, and verify the answer is a real knowledge response rather than a mock text.

**Acceptance Scenarios**:

1. **Given** a signed-in user opens the mobile assistant and asks a construction standards question, **When** the main agent calls the knowledge tool, **Then** the user receives a final knowledge answer in the conversation.
2. **Given** a signed-in user asks multiple different standards questions, **When** each question is routed to the knowledge tool, **Then** each response is answer-oriented and not the mock success text.

---

### User Story 2 - Hide Subagent Intermediate Events (Priority: P2)

As a mobile assistant user, I only see the final answer generated by the knowledge tool, and I never see intermediate subagent events such as internal streaming chunks or state transitions.

**Why this priority**: The product requires a clean, stable conversation experience and prevents leaking low-level tool execution details.

**Independent Test**: Trigger a knowledge query and verify the conversation stream and tool cards show only normal running/done status plus final answer, with no subagent intermediate payload visible.

**Acceptance Scenarios**:

1. **Given** the knowledge tool is processing a question, **When** intermediate subagent events occur internally, **Then** those events are not persisted into the main agent message state.
2. **Given** the frontend is receiving assistant stream updates, **When** a knowledge tool call is in progress, **Then** the frontend does not render any subagent intermediate event content.

---

### User Story 3 - Graceful Failure Messaging (Priority: P3)

As a user, if the external knowledge service fails, I still receive a clear and understandable fallback message and the conversation remains usable.

**Why this priority**: External services can fail, and the assistant must remain reliable under degraded conditions.

**Independent Test**: Use invalid credentials or force timeout conditions and verify the user sees clear failure feedback instead of a broken conversation.

**Acceptance Scenarios**:

1. **Given** the knowledge service is unavailable or unauthorized, **When** the tool is invoked, **Then** the user receives a clear error explanation in normal conversation tone.
2. **Given** a user cancels the in-flight conversation request, **When** the request is aborted, **Then** the external knowledge request is canceled promptly and no partial intermediate events are surfaced.

---

### Edge Cases

- The knowledge service returns no final answer content.
- The user sends a very short or ambiguous question that still routes to knowledge lookup.
- The user cancels the request while the subagent is still processing.
- The service returns a response that exceeds normal answer length and must still be presented as a single final answer.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST expose a renamed knowledge consultation tool in the main agent, replacing the current mock knowledge tool behavior.
- **FR-002**: The system MUST route construction standards/practices/compliance questions to the renamed knowledge consultation tool.
- **FR-003**: The tool MUST call the external Coze knowledge bot using runtime configuration values for token and bot identity from environment variables.
- **FR-004**: The tool MUST send the currently authenticated platform user identifier as the external request user identity.
- **FR-005**: The tool MUST return only the final answer text to the main agent, with no intermediate subagent event payload included in tool output.
- **FR-006**: The system MUST prevent intermediate subagent messages/states from being appended to main agent conversation state or emitted to frontend message stream.
- **FR-007**: The system MUST keep the existing frontend tool execution UX behavior (running/done indication) while still hiding subagent internals.
- **FR-008**: If the external call fails, times out, or is unauthorized, the tool MUST provide a readable failure message suitable for user-facing assistant response.
- **FR-009**: If the user aborts the conversation request, the tool MUST propagate cancellation to the external knowledge request.
- **FR-010**: The system MUST document newly required configuration keys in environment setup documentation.

### Key Entities *(include if feature involves data)*

- **KnowledgeQuestion**: A user-provided construction quality question submitted to the assistant for standards/practices lookup.
- **KnowledgeAnswer**: The finalized text answer returned from the external knowledge bot and passed back to the main agent.
- **KnowledgeToolCall**: A single invocation context that links authenticated platform user identity, the user question, and the final answer or failure message.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: In staged validation conversations, 95% or more knowledge-category questions return non-mock, domain-relevant answers.
- **SC-002**: In stream inspection for knowledge tool interactions, 0 intermediate subagent payload events are visible in user-facing conversation output.
- **SC-003**: In forced failure scenarios (invalid token, timeout, service unavailable), 100% of runs return user-readable failure responses without breaking the chat flow.
- **SC-004**: In cancellation tests, external knowledge requests stop within 2 seconds after user abort action.

## Assumptions

- Existing authenticated platform identity remains the single source for user identity passed to the external knowledge service.
- A single configured Coze bot is used for this feature scope and does not vary by project, role, or tenant.
- The feature scope is limited to returning final answer text only, without references or follow-up suggestion payloads.
- Existing assistant UI patterns for tool call progress remain unchanged, and only data visibility rules are adjusted.
- Conversation continuity relies on current platform behavior; no additional cross-request subagent session memory is introduced in this scope.
