# Feature Specification: 工单组件与状态更改动作

**Feature Branch**: `004-ticket-component-actions`
**Created**: 2026-04-12
**Status**: Draft
**Input**: User description: "实现阶段3，但是完全跳过变更记录的部分不做开发"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 查看工单详情 (Priority: P1)

质检员、施工方或管理员打开一个工单，查看工单的全部信息——基础信息（ID、状态、创建时间、发起人、所在项目、当前责任人）、问题描述详情（严重程度、专业类型、问题描述、详细位置、图片、问题详情）、以及复盘信息（问题归因、预防建议、是否已转化知识库）。信息按分组清晰展示，状态以 Badge 标签形式突出显示。

**Why this priority**: 查看是所有操作的基础，没有查看能力，后续编辑和状态操作无从谈起。这是整个工单模块的入口体验。

**Independent Test**: 从数据库读取真实工单数据渲染组件，验证所有字段正确展示、状态 Badge 正确渲染、图片正确展示、用户/项目小组件正确嵌入。

**Acceptance Scenarios**:

1. **Given** 一个状态为「待处理」的工单, **When** 用户以查看模式打开工单详情, **Then** 所有基础信息、详情信息正确展示，状态显示为「待处理」Badge，操作按钮根据当前用户身份和工单状态正确显示
2. **Given** 一个状态为「已完成」的工单, **When** 用户以查看模式打开, **Then** 问题归因和预防建议已填写并展示，「是否转化知识库」字段展示
3. **Given** 一个带有图片的工单, **When** 用户查看工单详情, **Then** 图片在详情区域正确展示

---

### User Story 2 - 编辑工单 (Priority: P2)

发起人或当前责任人点击「编辑」按钮，工单详情组件切换到编辑状态，可读写字段变为可编辑表单（严重程度、专业类型、问题描述、详细位置、图片、问题详情）。用户修改后点击「保存」，组件切回展示状态，更新后的内容即时反映。

**Why this priority**: 编辑是工单生命周期的常见操作，但依赖查看功能先完成。

**Independent Test**: 从数据库读取真实工单数据，点击编辑切换到编辑模式，修改字段后保存（写入数据库），验证组件切回展示模式并显示更新后的值。

**Acceptance Scenarios**:

1. **Given** 用户是工单发起人或当前责任人, **When** 用户点击「编辑」按钮, **Then** 组件切换到编辑状态，可读写字段变为表单输入
2. **Given** 用户在编辑状态, **When** 用户修改字段后点击「保存」, **Then** 组件切回展示状态，更新后的值正确反映
3. **Given** 用户不是工单发起人也不是当前责任人, **When** 用户查看工单, **Then** 「编辑」按钮不显示

---

### User Story 3 - 解决工单 (Priority: P3)

当前责任人（施工方）在待处理工单的详情页点击「解决」按钮，工单状态直接变为「已完成」。本阶段跳过问题归因和预防建议的填写流程。

**Why this priority**: 解决是工单最核心的状态流转操作，但需要查看和编辑功能先就绪。

**Independent Test**: 使用数据库中的待处理工单，当前用户身份为施工方（责任人），点击解决按钮（调用后端 API），验证工单状态在数据库中变为已完成。

**Acceptance Scenarios**:

1. **Given** 当前用户是工单责任人且工单状态为「待处理」, **When** 用户点击「解决」按钮, **Then** 工单状态直接变为「已完成」
2. **Given** 工单已变为「已完成」, **When** 用户刷新页面, **Then** 工单状态仍为「已完成」

---

### User Story 4 - 拒绝工单 (Priority: P4)

当前责任人（施工方）在待处理工单的详情页点击「拒绝」按钮，工单状态直接变为「已拒绝」。本阶段跳过拒绝原因的填写流程。

**Why this priority**: 拒绝是状态机的重要分支，但使用频率低于解决。

**Independent Test**: 使用数据库中的待处理工单，当前用户身份为施工方（责任人），点击拒绝按钮（调用后端 API），验证工单状态在数据库中变为已拒绝。

**Acceptance Scenarios**:

1. **Given** 当前用户是工单责任人且工单状态为「待处理」, **When** 用户点击「拒绝」按钮, **Then** 工单状态直接变为「已拒绝」
2. **Given** 工单已变为「已拒绝」, **When** 用户刷新页面, **Then** 工单状态仍为「已拒绝」

---

### User Story 5 - 重新打开工单 (Priority: P5)

发起人（质检员）在已完成或已拒绝的工单详情页点击「重新打开」按钮，工单状态直接变回「待处理」，责任人保持不变。本阶段跳过打开原因的填写流程。

**Why this priority**: 重新打开是闭环流程的关键操作，但仅在已关闭工单上触发，频率最低。

**Independent Test**: 使用数据库中的已完成/已拒绝工单，当前用户身份为发起人，点击重新打开按钮（调用后端 API），验证状态在数据库中变回待处理。

**Acceptance Scenarios**:

1. **Given** 当前用户是工单发起人且工单状态为「已完成」或「已拒绝」, **When** 用户点击「重新打开」按钮, **Then** 工单状态直接变回「待处理」，责任人不变
2. **Given** 工单已变回「待处理」, **When** 用户刷新页面, **Then** 工单状态仍为「待处理」

---

### Edge Cases

- 工单状态为「待处理」时，不显示「重新打开」按钮；状态为「已完成」或「已拒绝」时，不显示「解决」「拒绝」按钮
- 管理员身份时，所有满足状态前置条件的操作按钮均显示
- 编辑状态下切换工单或离开页面时，未保存的修改直接丢弃
- 工单无图片时，图片区域不展示占位符
- 状态更改操作进行中（API 请求期间）应防止重复点击

## Requirements *(mandatory)*

### Functional Requirements

**工单详情组件（展示模式）**

- **FR-001**: 工单详情组件 MUST 支持展示模式（view），以只读方式展示工单全部信息
- **FR-002**: 展示模式 MUST 按三组信息分区展示：基础信息、工单详情、工单复盘
- **FR-003**: 基础信息区 MUST 展示：工单 ID、工单状态（Badge 标签）、创建时间、发起人（用户小组件）、所在项目（项目小组件）、当前责任人（用户小组件）
- **FR-004**: 工单详情区 MUST 展示：严重程度、专业类型、问题描述、详细位置、图片（如有）、问题详情（如有）
- **FR-005**: 工单复盘区 MUST 展示：问题归因（如有）、预防建议（如有）、是否已转化知识库

**工单详情组件（编辑模式）**

- **FR-006**: 工单详情组件 MUST 支持编辑模式（edit），可读写字段变为表单输入
- **FR-007**: 编辑模式下基础信息区字段 MUST 为只读
- **FR-008**: 编辑模式下工单详情区 MUST 可编辑：严重程度（Enum 选择）、专业类型（Enum 选择）、问题描述（文本输入）、详细位置（文本输入）、图片（多图上传）、问题详情（长文本）
- **FR-009**: 编辑模式下工单复盘区 MUST 为只读（问题归因、预防建议在解决操作中填写；是否转化知识库不可编辑）
- **FR-010**: 保存编辑后 MUST 切换回展示模式并反映更新内容（数据持久化到数据库）

**操作按钮动态渲染**

- **FR-011**: 展示模式底部 MUST 根据当前用户身份和工单状态动态显示/隐藏操作按钮
- **FR-012**: 「编辑」按钮 MUST 在当前用户是发起人或当前责任人时显示
- **FR-013**: 「解决」按钮 MUST 在当前用户是当前责任人且工单状态为「待处理」时显示
- **FR-014**: 「拒绝」按钮 MUST 在当前用户是当前责任人且工单状态为「待处理」时显示
- **FR-015**: 「重新打开」按钮 MUST 在当前用户是发起人且工单状态为「已完成」或「已拒绝」时显示
- **FR-016**: 管理员 MUST 拥有所有操作权限，以上按钮在满足状态前置条件时均显示

**状态更改动作**

- **FR-017**: 解决操作点击后 MUST 直接调用后端 API 将工单状态变为「已完成」（本阶段不要求填写问题归因和预防建议）
- **FR-018**: 拒绝操作点击后 MUST 直接调用后端 API 将工单状态变为「已拒绝」（本阶段不要求填写拒绝原因）
- **FR-019**: 重新打开操作点击后 MUST 直接调用后端 API 将状态变回「待处理」，责任人不变（本阶段不要求填写打开原因）
- **FR-020**: 状态更改操作进行中 MUST 禁用操作按钮，防止重复提交

**工单后端 API**

- **FR-021**: 系统 MUST 提供工单 CRUD 后端 API（创建、读取、更新）
- **FR-022**: 系统 MUST 提供工单状态机操作 API（解决、拒绝、重新打开）
- **FR-023**: 工单写入 MUST 仅通过后端 API，不允许直接调用数据库写入
- **FR-024**: 状态机操作 API MUST 校验操作权限和前置状态条件

**工单数据表**

- **FR-025**: 系统 MUST 创建工单数据表（tickets）及 RLS 策略，登录用户只可读取当前身份所在项目下的工单

### Key Entities

- **工单（Ticket）**: 核心业务实体，包含基础信息（ID、状态、创建时间、发起人、项目、责任人）、详情信息（严重程度、专业类型、问题描述、详细位置、图片、问题详情）、复盘信息（问题归因、预防建议、是否转化知识库）。状态有三种：待处理、已完成、已拒绝。
- **用户（User）**: 通过用户小组件在工单详情中内联展示（头像+部门+姓名），同时决定操作按钮的显示逻辑。
- **项目（Project）**: 通过项目小组件在工单详情中展示，工单归属于项目。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能在 5 秒内理解工单详情页的全部信息结构和分组
- **SC-002**: 工单详情组件正确展示所有字段，信息分组清晰，不同身份用户看到正确的操作按钮
- **SC-003**: 所有状态更改操作（解决、拒绝、重新打开）在满足条件时点击即完成，在不满足条件时按钮不显示
- **SC-004**: 编辑保存后工单信息即时更新，组件正确切回展示模式

## Assumptions

- **直接对接真实数据库**——不使用 mock 数据，工单数据表和后端 API 全部使用真实 Supabase 数据库
- **变更记录（ticket_logs）完全不在本阶段开发范围内**——后端 API 不写 ticket_logs，前端不展示变更记录，ticket_logs 表也不在本阶段创建
- **指派他人操作本阶段不实现**——按钮不渲染，后端 API 不提供此端点
- **状态更改动作暂不要求补充输入**——解决不填问题归因/预防建议，拒绝不填原因，重新打开不填原因；后续阶段补充
- 工单详情组件为双端共用组件，本阶段直接在移动端工单详情页 `/mobile/tickets/[id]` 中开发和测试；PC 端抽屉集成留待阶段 6
- 图片上传在编辑模式中使用占位 UI（选择文件的交互），实际存储后续对接
- 工单创建由 Agent 模块触发（阶段 5），本阶段不涉及创建流程的前端 UI，但后端 API 需支持创建操作
- 管理员拥有所有操作权限，实现时统一处理
- 需要为工单表准备种子数据，以便开发和测试
