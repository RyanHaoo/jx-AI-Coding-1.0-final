# Feature Specification: Agent 工单查询 MCP 接入

**Feature Branch**: `006-agent-ticket-query-mcp`  
**Created**: 2026-04-18  
**Status**: Draft  
**Input**: User description: "实现 Agent 的工单查询 MCP 开发和接入（仅 query_ticket，使用当前用户 Supabase session 并确保 RLS）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 当前身份下查询待处理工单 (Priority: P1)

质检员或施工方在智能助手中提出工单查询问题（例如“有哪些待处理工单”），系统应返回当前身份所属项目下的工单摘要列表，帮助用户快速了解待办问题。

**Why this priority**: 这是 Agent 查询能力的核心主路径，直接决定用户是否能通过自然语言完成工单检索。

**Independent Test**: 在已登录并已选择身份的情况下，发送“有哪些待处理工单”，可独立验证是否返回当前项目的工单列表且字段完整。

**Acceptance Scenarios**:

1. **Given** 用户已登录并已选择项目身份，**When** 用户发送“有哪些待处理工单”，**Then** 系统返回该项目内待处理工单列表，包含工单关键信息摘要
2. **Given** 用户已登录并已选择项目身份，**When** 用户发送“查询工单”，**Then** 系统触发工单查询工具并返回最多默认条数的结果

---

### User Story 2 - 按条件筛选工单 (Priority: P2)

用户在查询时附带筛选条件（如状态、严重程度、专业类型、关键字、数量上限），系统应按条件返回结果，便于用户定位目标工单。

**Why this priority**: 在主路径可用后，筛选能力决定结果可用性和对话效率，能显著降低用户二次追问成本。

**Independent Test**: 发送“查询严重级别高且未关闭的工单”，可独立验证结构化筛选是否生效。

**Acceptance Scenarios**:

1. **Given** 用户已登录且有项目身份，**When** 用户发起带状态与严重程度的查询，**Then** 返回结果仅包含符合条件的工单
2. **Given** 用户将返回上限设置为超出系统允许值，**When** 系统执行查询，**Then** 返回结果受系统上限约束并提示已截断

---

### User Story 3 - 越权与未登录保护 (Priority: P1)

系统必须确保查询结果只来自当前身份可访问的数据范围；未登录或会话失效时，不返回项目数据并提示重新登录。

**Why this priority**: 数据权限是该能力的安全底线，优先级与主路径同级。

**Independent Test**: 使用同一账号切换不同项目身份分别查询，可独立验证返回集合变化且不混入其他项目数据；退出登录后查询应失败并给出明确提示。

**Acceptance Scenarios**:

1. **Given** 用户从 A 项目身份切换到 B 项目身份，**When** 用户执行同类查询，**Then** 返回结果仅来自 B 项目且不包含 A 项目工单
2. **Given** 用户会话缺失或已过期，**When** 用户触发工单查询，**Then** 系统不返回工单数据并提示“请重新登录后再试”

---

### Edge Cases

- 查询无匹配数据时，返回空列表并给出“未找到符合条件工单”的明确反馈
- 查询条件仅包含关键字且关键字为空白时，按未提供关键字处理，不应报错
- 结果总量超过单次返回上限时，返回前 N 条并标记结果已截断
- 会话存在但访问凭证失效时，返回受控错误，不暴露底层异常细节

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供一个用于工单查询的单一工具能力，供 Agent 在“查询工单”意图下调用
- **FR-002**: 系统 MUST 支持以下可选筛选条件：状态、严重程度、专业类型、关键字、返回数量上限
- **FR-003**: 系统 MUST 在未显式指定返回数量上限时使用默认值 10
- **FR-004**: 系统 MUST 将单次查询返回数量限制在 50 条以内
- **FR-005**: 系统 MUST 在返回结果中仅包含工单摘要字段：`id`、`status`、`severity`、`description`、`location`、`created_at`、责任人姓名、项目名称
- **FR-006**: 系统 MUST 不在查询结果中返回大文本或大体积字段（如图片、详细过程、根因分析）
- **FR-007**: 系统 MUST 使用当前登录用户的有效会话凭证执行查询，且凭证不向模型暴露
- **FR-008**: 系统 MUST 依赖既有数据权限策略确保仅返回当前身份可访问的数据
- **FR-009**: 系统 MUST 在会话缺失或无效时返回受控失败结果，并引导用户重新登录
- **FR-010**: 系统 MUST 在结果被截断时提供可识别标记，便于 Agent 提示用户收敛筛选条件
- **FR-011**: 系统 MUST 保持其他现有 mock 工具行为不变（知识查询与创建工单不在本次范围）
- **FR-012**: 系统 MUST 默认按创建时间倒序返回结果

### Key Entities *(include if feature involves data)*

- **Ticket Query Input**: 用户查询的结构化条件集合，包含可选过滤字段与返回条数限制
- **Ticket Summary Item**: 单条工单摘要视图，包含最小必要字段，用于对话上下文展示
- **Session-Scoped Access Context**: 与当前登录用户绑定的访问上下文，决定本次查询可见数据范围

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在已登录且有可访问工单的数据集上，90% 以上“查询工单”请求可在一次对话内返回有效结果
- **SC-002**: 在同账号跨项目身份切换测试中，查询结果跨项目数据泄漏率为 0
- **SC-003**: 单次工单查询端到端响应时间在常规数据量场景下不超过 3 秒
- **SC-004**: 当请求超出返回上限时，100% 的响应包含截断标记，且返回条数不超过系统上限

## Assumptions

- 当前系统已具备稳定的登录态与身份切换能力，查询能力复用该会话上下文
- 当前数据权限策略已覆盖“仅可读取所属项目工单”的访问边界
- 首版本仅覆盖查询能力，不包含分页、复杂排序与扩展统计能力
- 查询结果用于 Agent 对话总结展示，优先保证精简和可读性而非全量字段输出

## Out of Scope

- 不新增除 `query_ticket` 之外的 MCP 工具
- 不实现 MCP 的 HTTP 或 SSE 传输方式
- 不改造知识查询与创建工单流程
- 不在本规格阶段引入新的监控与追踪平台接入
