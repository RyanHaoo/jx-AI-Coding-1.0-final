# Quickstart: 移动端工单界面验收

**Feature**: 005-mobile-ticket-detail  
**Date**: 2026-04-18

用于开发者在本地验证本 spec 的 5 个用户故事。验收清单以 spec.md 的 FR-XXX / SC-XXX 反向映射。

---

## 前置准备

```bash
npm install
npm run dev   # http://localhost:3000
```

测试账号（见 `README.md`，密码统一 `test123456`）：

| 工号 | 身份 | 预期数据 |
|------|------|----------|
| `QC001`（张质检） | 质检员 × 翡翠湾花园 | 至少 2 条其发起的工单 |
| `SV001`（李施工） | 施工方 × 翡翠湾花园 | 至少 1 条其负责的待处理工单 |

## 场景 1：未登录重定向（对应 FR-017 / SC-005）

1. 打开浏览器隐身窗，访问 `http://localhost:3000/mobile/tickets`。
2. **预期**：被重定向到 `/login?redirect=/mobile/tickets`。
3. 用 `QC001` 登录并选择身份。
4. **预期**：回跳到 `/mobile/tickets`。

## 场景 2：列表页 Tab 过滤 + 紧急置顶（对应 US2 / FR-002..005）

1. 以 `QC001` 身份进入 `/mobile/tickets`。
2. 默认 Tab 为"待处理"。
3. **预期**：
   - 列表仅显示 `status="待处理"` 的工单。
   - 严重程度为"紧急"的工单排在最前面，且卡片带红色边框。
   - 非紧急的按创建时间倒序。
4. 点击"已结束"Tab。
5. **预期**：列表切换为 `已完成 + 已拒绝` 的并集；紧急置顶规则同样生效。
6. 点击"全部"Tab。
7. **预期**：所有状态工单都显示。

## 场景 3：进入详情 + 返回（对应 US1 / FR-007 / FR-009）

1. 继续场景 2，点击任一工单卡片。
2. **预期**：跳转到 `/mobile/tickets/<id>`，顶栏标题变为"工单详情"，左侧为返回箭头。
3. 查看详情：
   - 基础信息完整，状态 Badge 正确。
   - 有图片时图片正确显示。
   - 底部操作按钮符合 004 规则（`QC001` 为发起人 → 应显示"编辑"或"重新打开"）。
4. 点击返回箭头。
5. **预期**：回到 `/mobile/tickets`，Tab 状态保持（浏览器自带的 state 恢复；MVP 不强保证滚动位置）。

## 场景 4：编辑工单（对应 US4 / FR-013）

1. 以 `QC001`（发起人）或 `SV001`（责任人）身份打开一条"待处理"工单详情。
2. 点击"编辑"。
3. **预期**：详情组件切换为编辑模式，可读写字段出现表单控件。
4. 修改"问题描述"或"详细位置"，点击"保存"。
5. **预期**：组件切回展示模式，显示新值；刷新页面后新值仍在。

## 场景 5：状态流转（对应 US3 / FR-014 / FR-015）

1. 以 `SV001`（责任人）身份打开一条"待处理"工单详情。
2. 点击"解决"。
3. **预期**：状态 Badge 变为"已完成"；"解决 / 拒绝"按钮消失；无需手动刷新。
4. 返回列表，切到"已结束"Tab。
5. **预期**：该工单出现在列表中。
6. 切到 `QC001`（发起人），打开该已完成工单，点击"重新打开"。
7. **预期**：状态回到"待处理"；返回列表后在"待处理"Tab 中可见。
8. 在请求进行中时重复点击按钮。
9. **预期**：按钮被禁用，不产生重复请求（FR-015）。

## 场景 6：404 / 无权访问（对应 FR-012）

1. 以 `QC001`（翡翠湾花园）身份，在地址栏手动访问一条属于"星河产业园"的工单 URL（或任一不存在的 ID）。
2. **预期**：页面展示 404 / "工单不存在或无权访问" 提示，并提供返回列表链接。

## 场景 7：空状态（对应 FR-008）

1. 切换到一个没有任何工单的项目（如新建一个种子项目且不写工单）。
2. 进入 `/mobile/tickets`。
3. **预期**：列表区显示"暂无工单"提示。
