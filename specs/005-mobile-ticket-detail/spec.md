# Feature Specification: 移动端工单界面（列表 + 详情组件接入）

**Feature Branch**: `005-mobile-ticket-detail`
**Created**: 2026-04-18
**Status**: Draft
**Input**: User description: "完成工单详情组件开发，并接入移动端的工单界面"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 打开移动端工单详情页查看工单 (Priority: P1)

质检员、施工方或管理员在移动端通过侧边目录或列表项进入某条工单的详情页（`/mobile/tickets/:id`）。页面顶栏提供返回按钮；页面主体由**工单详情组件**（已在 004 中定义）承载，展示基础信息、工单详情、工单复盘三组字段。操作按钮根据当前用户身份和工单状态动态渲染。

**Why this priority**: 详情页是移动端工单模块的核心入口，承载了工单生命周期内最主要的查看、编辑与状态流转动作。若详情页不可用，整个移动端工单流程就断裂。

**Independent Test**: 使用种子数据中任一工单，在已登录并选定身份的前提下访问 `/mobile/tickets/:id`，能看到完整的工单详情视图、正确的 Badge/小组件、以及与当前身份匹配的操作按钮集合；点击返回能回到工单列表页。

**Acceptance Scenarios**:

1. **Given** 当前用户为工单发起人且工单状态为「待处理」, **When** 用户访问 `/mobile/tickets/:id`, **Then** 页面展示完整工单信息，且底部显示「编辑」「重新打开」按钮的逻辑与 004 规格一致（此情境下仅显示「编辑」）
2. **Given** 当前用户为工单当前责任人且工单状态为「待处理」, **When** 用户访问详情页, **Then** 底部显示「编辑」「解决」「拒绝」按钮
3. **Given** 工单存在图片, **When** 用户查看详情页, **Then** 图片正确加载展示
4. **Given** 用户点击顶栏返回按钮, **When** 页面切出, **Then** 返回到 `/mobile/tickets` 列表页，并保留上一次的筛选状态

---

### User Story 2 - 在移动端工单列表中按状态筛选工单 (Priority: P2)

用户打开 `/mobile/tickets` 进入工单列表页，页面顶部提供「待处理 / 已结束 / 全部」三个 Tab。切换 Tab 时列表即时过滤并重新排序，紧急工单始终置顶并以红色高亮边框突出。点击任一列表项可跳转到该工单的详情页。

**Why this priority**: 列表页是进入详情页的主要入口；Tab 筛选是最常用的查阅动作，直接决定用户能否快速定位关心的工单。

**Independent Test**: 准备多种状态与严重程度的工单种子数据，用户切换 Tab 能看到列表在前端立即过滤；紧急工单无论 Tab 如何始终置顶并带红色边框；点击列表项能进入对应详情页。

**Acceptance Scenarios**:

1. **Given** 当前身份所在项目有多条工单, **When** 用户切换到「待处理」Tab, **Then** 列表仅展示状态为「待处理」的工单
2. **Given** 用户切换到「已结束」Tab, **When** 列表刷新, **Then** 展示状态为「已完成」或「已拒绝」的工单
3. **Given** 列表中存在严重程度为「紧急」的工单, **When** 用户查看任一 Tab, **Then** 紧急工单位于列表顶部并带红色高亮边框
4. **Given** 用户点击一条列表项, **When** 页面跳转, **Then** 进入 `/mobile/tickets/:id` 详情页

---

### User Story 3 - 在移动端详情页完成工单状态流转 (Priority: P3)

施工方（责任人）或管理员在详情页点击「解决」或「拒绝」按钮，工单状态直接变更为「已完成」或「已拒绝」（本阶段不要求补充输入）；发起人（质检员）或管理员在已关闭工单的详情页点击「重新打开」，工单状态回到「待处理」。状态变更成功后，详情页即时反映新的状态 Badge，并更新可见的操作按钮集合。

**Why this priority**: 状态流转是工单业务闭环的关键；用户故事 1 已保证界面承载，本用户故事把状态机操作在移动端打通。

**Independent Test**: 登录不同身份用户，在同一条「待处理」工单详情页分别点击「解决」「拒绝」；在已关闭工单上以发起人身份点击「重新打开」，验证每次操作后数据库状态与界面 Badge/按钮同步更新，且在 API 请求期间按钮处于禁用状态。

**Acceptance Scenarios**:

1. **Given** 当前用户为责任人且工单为「待处理」, **When** 用户点击「解决」, **Then** 工单状态变为「已完成」，详情页 Badge 与按钮集合即时更新
2. **Given** 当前用户为责任人且工单为「待处理」, **When** 用户点击「拒绝」, **Then** 工单状态变为「已拒绝」，详情页同步刷新
3. **Given** 当前用户为发起人且工单为「已完成」或「已拒绝」, **When** 用户点击「重新打开」, **Then** 工单状态回到「待处理」且责任人不变
4. **Given** 状态变更 API 请求进行中, **When** 用户重复点击按钮, **Then** 按钮保持禁用状态，不产生重复请求

---

### User Story 4 - 在移动端详情页编辑工单可写字段 (Priority: P4)

发起人、当前责任人或管理员在详情页点击「编辑」按钮，工单详情组件切换到编辑模式；用户修改严重程度、专业类型、问题描述、详细位置、图片、问题详情后点击「保存」，组件切回展示模式并显示更新后的值。

**Why this priority**: 编辑能力让用户在提交后仍能补充/修正信息，但使用频率低于查看与状态流转，优先级次之。

**Independent Test**: 使用任意工单，以有编辑权限的身份进入详情页，点击「编辑」后修改若干字段，点击「保存」后验证页面切回展示模式并显示新值；刷新页面后值仍然保留（来自数据库）。

**Acceptance Scenarios**:

1. **Given** 当前用户为发起人或当前责任人, **When** 用户点击「编辑」, **Then** 详情组件切换到编辑模式，可读写字段变为表单控件
2. **Given** 用户在编辑模式下修改字段, **When** 用户点击「保存」, **Then** 组件切回展示模式并显示新值，数据已持久化
3. **Given** 用户不是发起人也不是当前责任人（也非管理员）, **When** 用户查看详情页, **Then** 「编辑」按钮不显示

---

### User Story 5 - 列表页的加载、空、错误与鉴权提示 (Priority: P5)

工单列表页和详情页在数据加载过程中提供明确的加载态；当前筛选/路由下无数据时显示空状态；网络或权限错误时显示可理解的错误提示并提供重试或返回入口；未登录访问时自动跳转到 `/login?redirect=...`。

**Why this priority**: 这些态直接影响用户第一印象与容错体验，但不是核心路径；在核心故事成立后再完善。

**Independent Test**: 模拟慢速网络、空结果、API 错误、未登录访问四种情况，验证界面分别呈现加载态、空状态、错误态以及登录跳转。

**Acceptance Scenarios**:

1. **Given** 当前身份所在项目无任何工单, **When** 用户访问列表页, **Then** 列表区显示空状态插图与提示文字（如「暂无工单」）
2. **Given** 工单详情接口返回 404 或 403, **When** 页面加载完成, **Then** 展示「工单不存在或无权访问」提示并提供返回列表的按钮
3. **Given** 用户未登录, **When** 用户访问 `/mobile/tickets` 或 `/mobile/tickets/:id`, **Then** 浏览器被跳转到 `/login?redirect=<原路径>`

---

### Edge Cases

- 用户从详情页使用浏览器「后退」返回列表，列表应保持上次 Tab 与滚动位置（尽力而为，不作强保证）
- 工单无图片时，详情页图片区域不渲染占位符；列表项卡片不显示左侧预览图
- 用户在编辑模式下切走/跳转其他路由，未保存修改直接丢弃，不做确认拦截（MVP 范围）
- 工单详情页加载时若工单被其他人状态变更，本次加载仍以返回值为准（不做实时推送）
- 状态变更或编辑保存期间页面不应允许重复点击按钮

## Requirements *(mandatory)*

### Functional Requirements

**工单列表页（`/mobile/tickets`）**

- **FR-001**: 列表页 MUST 仅展示当前用户当前身份所在项目下的工单
- **FR-002**: 列表页 MUST 在顶部提供「待处理 / 已结束 / 全部」三个 Tab；「已结束」MUST 覆盖「已完成」与「已拒绝」两种状态
- **FR-003**: 切换 Tab MUST 即时过滤当前列表，不需要整页刷新
- **FR-004**: 紧急工单（严重程度=紧急）MUST 始终置顶显示，并带红色高亮边框，不受 Tab/排序影响
- **FR-005**: 非紧急工单 MUST 按创建时间倒序（最新优先）默认排序
- **FR-006**: 列表项 MUST 展示工单编号、状态 Badge、严重程度 Badge、截断后的问题描述、详细位置、创建时间、当前责任人（用户小组件）；工单有图片时 MUST 在卡片左侧展示首图预览
- **FR-007**: 点击列表项 MUST 跳转到对应 `/mobile/tickets/:id` 详情页
- **FR-008**: 列表加载过程中 MUST 显示加载态；筛选结果为空时 MUST 显示空状态插图与提示文字；接口错误时 MUST 显示错误提示与重试入口

**工单详情页（`/mobile/tickets/:id`）**

- **FR-009**: 详情页 MUST 有顶栏返回按钮，点击后回到 `/mobile/tickets`
- **FR-010**: 详情页主体 MUST 由 **工单详情组件**（spec 004）承载；详情组件内部的字段展示、模式切换、操作按钮规则完全遵循 004 规格
- **FR-011**: 详情页 MUST 根据路由参数 `id` 调用工单详情后端 API 加载数据；加载中 MUST 显示骨架屏或加载态
- **FR-012**: 工单不存在或当前用户无权访问（不属于当前身份所在项目）时 MUST 显示 404/无权访问提示并提供返回列表按钮

**状态变更与编辑（复用 004 组件能力）**

- **FR-013**: 详情页 MUST 支持组件内的编辑模式切换；保存后组件 MUST 切回展示模式并反映更新值
- **FR-014**: 详情页 MUST 支持「解决 / 拒绝 / 重新打开」三种状态变更动作；动作完成后 MUST 即时刷新状态 Badge 与按钮集合，无需用户手动刷新页面
- **FR-015**: 状态变更或保存请求进行中 MUST 禁用相关按钮，防止重复提交

**移动端通用布局与鉴权**

- **FR-016**: `/mobile/tickets` 与 `/mobile/tickets/:id` MUST 位于移动端共享布局下（顶栏 + 可通过汉堡菜单打开的侧边目录）
- **FR-017**: 未登录用户访问上述路由 MUST 被自动重定向到 `/login?redirect=<原路径>`；登录后 MUST 回跳到原路径
- **FR-018**: 尚未完成身份选择的已登录用户访问 MUST 触发身份选择流程

### Key Entities

- **工单（Ticket）**：核心业务实体，沿用 004 中定义。列表页按当前身份所在项目过滤；详情页按 ID 读取并用于驱动工单详情组件
- **当前身份（Active Identity）**：当前登录用户选定的 `用户 × 项目 × 角色` 三元组；决定列表页过滤范围与详情页操作按钮可见性
- **用户（User）** 与 **项目（Project）**：在列表项与详情组件中以小组件形式内联展示，沿用已有组件实现

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户进入移动端工单列表后，能在 10 秒内找到某条目标工单并进入其详情页
- **SC-002**: 对同一条工单，不同身份的用户在详情页看到的可执行操作按钮集合与 004 规则完全一致（按情境抽样 100%）
- **SC-003**: 切换 Tab 后列表过滤结果在 1 秒内可视化更新（MVP 不作严格性能验收，仅要求可感知无明显卡顿）
- **SC-004**: 所有状态变更动作（解决/拒绝/重新打开）完成后，详情页的状态 Badge 与按钮集合即时与数据库状态一致，刷新页面结果不变
- **SC-005**: 未登录访问任一移动端工单路由 100% 触发正确的登录重定向并能成功回跳

## Assumptions

- **工单详情组件已在 spec 004 完成**：本 spec 不重复定义组件内部字段、操作按钮权限与状态机规则，完全遵循 004
- **后端工单 API 已在 spec 004 完成**：包括详情查询、编辑保存、解决/拒绝/重新打开三个状态动作端点；本 spec 复用
- **列表页不做分页**：MVP 阶段直接 limit 50 拉取当前项目工单
- **状态变更不需要补充输入**：MVP 阶段解决/拒绝/重新打开直接执行，不弹出表单（沿用 004 的 clarification 决定）
- **图片上传交互为占位**：列表与详情展示已上传图片；编辑模式下图片上传交互为占位 UI，实际存储留待后续阶段
- **工单列表责任人过滤（我负责的/我提出的）与排序切换**不在本 spec 范围内；作为后续增强（progress.md 阶段 4 最后一项）
- **不做实时推送**：工单在别处被修改后，用户当前页面不会自动刷新；下次进入或手动刷新时生效
- **不做编辑离开确认**：编辑模式下切换路由直接丢弃未保存内容，MVP 阶段不拦截
