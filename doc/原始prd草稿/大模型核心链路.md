# 大模型核心链路

## 1. 架构总览

```
用户(多角色) ↔ 前端UI ↔ 主 Agent ↔ 知识获取子Agent（外部api形式） + 工单查询MCP + create_ticket tool_call(HITL) + 对话数据库
```

**关键设计要点：**
- **单一身份：** 用户登录时固定单一身份（项目+角色），不存在运行中多身份场景
- **多模态：** 用户可带图片输入，主 Agent 需将图片信息纳入上下文
- **主 Agent 是唯一对话出口：** 子 Agent 不直接与用户对话，只返回知识文本给主 Agent
- **知识子 Agent 多轮检索：** 信息不足时自动追加查询，而非一次就返回
- **Human-In-The-Loop 工单创建：** Agent 不直接创建工单，而是通过 `create_ticket` tool_call 通知前端渲染工单卡片，由用户确认补充信息后前端调用后端 API 完成创建（详见第3节）

---

## 2. 主 Agent

**职责：** 接收用户输入，判断意图，调度子 Agent 或工具，与用户多轮对话直到任务完成。

### 2.1 上下文

- **System Prompt**（见下方框架）
- **对话历史**：包括工具调用等所有信息，按一问一答算仅保留 6 轮
- **当前用户输入**（文本 + 图片）

### 2.2 System Prompt 框架

```
你是「建筑施工质检情报员」的智能助手，服务于建筑施工质检场景。

## 当前用户身份
- 姓名：{user_name}
- 部门：{department}
- 角色：{role}（质检员/施工方/管理员）
- 所在项目：{project_name}（{project_type}）

## 你的能力
1. 知识问答：回答建筑施工规范、工艺标准相关问题
2. 工单查询：查询当前项目下的工单状态和处理进度
3. 创建工单：帮助质检员快速创建质检报事工单

## 行为规则
- 始终用中文回复
- 只能操作当前项目范围内的数据
- 创建工单时，必须从用户输入中提取：问题描述、详细位置、严重程度、专业类型，如有图片则纳入上下文
- 如果用户意图不明确，优先追问关键信息而非猜测
- 对于非建筑施工领域的问题，礼貌告知超出服务范围
- 回答时引用相关规范条文，增强可信度
```

### 2.3 意图路由规则

| 意图 | 判断依据 | 路由到 |
|------|----------|--------|
| 知识问答 | 用户询问施工规范、工艺标准、设计要求等知识性问题 | 调用知识子 Agent API |
| 工单查询 | 用户询问某个工单进度、状态，或提到工单编号 | 调用 queryTicket MCP |
| 创建工单 | 用户描述现场质量问题（带/不带图片），要求报事 | 调用 create_ticket tool_call |
| 混合意图 | 同时涉及知识查询和报事 | 先执行知识查询，再根据用户追问决定是否创建工单 |
| 闲聊/超范围 | 与建筑施工质检无关的对话 | 礼貌告知超出范围 |

### 2.4 内部功能模块

- **智能意图识别** — 解析用户输入意图，按上述规则路由
- **多轮对话管理** — 维护对话状态，管理追问与上下文 ↔ 对话数据库
- **工具调度** — 调用 queryTicket MCP 和 create_ticket tool_call
- **stream 消息广播** — 向前端 UI 推送流式响应
- **外部知识库接入** — 调用知识获取子 Agent

## 3. 工具定义

### 3.1 create_ticket Tool Call（Human-In-The-Loop）

主 Agent 识别到用户有创建工单的意图后，返回 `create_ticket` tool_call。**Agent 不直接创建工单**，而是通过 tool_call 通知前端渲染工单创建卡片，由用户确认并补充信息（如责任人选择）后，前端调用后端 API 完成实际创建。创建成功后，前端将结果返回给 Agent 继续对话。

当前端收到此 tool_call 时，渲染工单创建卡片供用户确认提交。

```json
{
  "name": "create_ticket",
  "description": "创建一条质检工单。需要提供问题描述、位置、严重程度、专业类型等信息。",
  "parameters": {
    "type": "object",
    "properties": {
      "description": {
        "type": "string",
        "description": "问题描述，简要说明发现了什么问题"
      },
      "location": {
        "type": "string",
        "description": "详细位置，如楼栋-楼层-区域"
      },
      "severity": {
        "type": "string",
        "enum": ["轻微", "一般", "严重", "紧急"],
        "description": "严重程度"
      },
      "specialty_type": {
        "type": "string",
        "enum": ["建筑设计专业", "结构专业", "给排水专业"],
        "description": "专业类型"
      },
      "detail": {
        "type": "string",
        "description": "问题详情补充（可选）"
      },
      "images": {
        "type": "array",
        "items": { "type": "string" },
        "description": "图片URL列表（可选）"
      }
    },
    "required": ["description", "location", "severity", "specialty_type"]
  }
}
```

**前端渲染规则：**
- 将 tool_call parameters 预填到工单创建表单
- 额外增加「责任人」下拉选择（必填，从当前项目的施工方用户中选择）
- 用户可修改任何字段后再提交
- 提交后前端调用 `POST /api/tickets` 创建工单
- 创建成功后将工单信息作为 tool_call result 返回给 Agent

### 3.2 queryTicket MCP Tool（只读查询）

工单查询通过 MCP 协议实现，主 Agent 直接调用即可获取工单数据。与 create_ticket 不同，这是纯读操作，不需要 Human-In-The-Loop。

使用 LangChain 的 stdio MCP 实现，每次 Agent 开始都启动一个 MCP 服务器，结束后自动停止

```json
{
  "name": "queryTicket",
  "description": "查询当前项目下的工单信息，支持按状态和关键词筛选",
  "parameters": {
    "type": "object",
    "properties": {
      "ticket_id": {
        "type": "integer",
        "description": "工单ID，精确查询指定工单（可选）"
      },
      "status": {
        "type": "string",
        "enum": ["待处理", "已完成", "已拒绝"],
        "description": "按状态筛选（可选）"
      },
      "keyword": {
        "type": "string",
        "description": "关键词搜索，匹配问题描述（可选）"
      },
      "assigned_to_me": {
        "type": "boolean",
        "description": "是否只查我负责的工单（默认false）"
      }
    }
  }
}
```

**返回格式：**
```json
{
  "tickets": [
    {
      "id": 1,
      "status": "待处理",
      "severity": "严重",
      "specialty_type": "结构专业",
      "description": "3#楼标准层梁底裂缝",
      "location": "东区一期-3#住宅楼-标准层",
      "assignee": "张工",
      "created_at": "2026-03-15T10:30:00"
    }
  ],
  "total": 1
}
```

**权限：** 继承当前用户身份，只能查询当前项目下的工单，不能编辑。

> **注意：** 工单的写操作（创建、解决、拒绝、指派、重新打开、编辑）统一通过 Next.js 后端 API 完成，不走 MCP 通道。MCP 仅用于只读查询。

## 4. 知识获取子 Agent

**职责：** 在多种知识源中查询输入的主题，并返回详细的知识文本。

已迁移到外部 Agent 托管平台，此处忽略内部实现，直接调用 API URL 即可。

### 4.1 调用规范

主 Agent 应传入**精练的待查询主题**，而非复述用户提问。

规则：
- 在前序会话中已经检索清楚的主题不要再次传入
- 多个议题使用两个换行隔开
- 指代消解：将代词替换为具体实体名词
- 补全上下文：从前序对话中提取省略的背景信息
- 意图提纯：去除口语化前缀和语气词，保留核心检索词
- 不回答问题，只重构查询

示例：用户输入「做学校设计的时候，该怎么设计门框和窗宽？要考虑哪些采光因素？」，主 Agent 应传入：
```
学校建筑设计中门框、窗宽的设计尺寸要求，门窗数量设置原则

学校建筑设计中门窗设置对采光的影响
```

### 4.2 Query Rewriter 提示词

以下提示词由主 Agent 内部调用，用于将用户原始提问改写为标准检索查询：

```
# Role
你是一个专业的查询改写专家（Query Rewriter），专注于对话系统和信息检索优化。

# Objective
你的任务是根据用户的"历史对话（Chat History）"，将用户的"当前提问（Current Query）"改写为一个信息完整、意图明确、且可独立用于搜索引擎或向量数据库高精度检索的标准查询语句（Standalone Query）。

# Guidelines
1. 指代消解：如果当前提问中包含代词（如"它"、"那个问题"、"这个功能"）或隐式指代，必须根据上下文将其替换为具体的实体名词或业务术语。
2. 补全上下文：如果当前提问省略了核心背景信息（如特定的系统名称、项目、产品或前置条件），请从历史对话中精准提取并补充完整。
3. 保持原意且不回答：改写后的语句必须严格忠于原始意图。你的任务只是"重构提问"，绝对不要去回答该问题，也不要发散或添加未提及的假设。
4. 意图提纯：去除无意义的口语化前缀或语气词（如"请问"、"那个"、"帮我查一下"），提取最核心的检索词或操作意图，使其更符合知识检索或任务创建的匹配逻辑。如果当前提问在脱离上下文时本身已经是完整的，则保持原样输出。
```

## 5. 用户流程

### 流程 A：专业知识问答

> 用户："上海的住宅阳台栏杆需要多高"

```
用户提问 → 主Agent意图识别 → 调用知识子Agent → 整理回答用户
```

### 流程 B：工单信息查询

> 用户："3号楼那个工单进度怎么样了"

```
用户提问 → 主Agent意图识别 → 调用queryTicket MCP → 向用户汇报进度
```

### 流程 C：质检员报事创建工单

> 用户："一楼楼板开裂[image-url]"

```
用户报事 → 主Agent识别报事参数 → create_ticket tool_call → 前端渲染创建卡片（含责任人选择）→ 用户确认提交 → 前端返回ticket_id → Agent告知创建完成
```

### 流程 D：知识 + 工单联动

> 用户："[施工图栏杆节点截图-url]这个高度合理吗？"

```
用户提问 → 主Agent多模态理解 → 调用知识子Agent → 回答规范判定

用户追问"帮我创建一个工单" → 主Agent整理上下文 → create_ticket tool_call → 前端确认提交 → Agent告知创建完成
```
