### 工单详情组件
- 两个状态:编辑 / 展示

| 操作动作 | 触发角色 | 前置状态 | 界面交互 (UI) | 业务逻辑与数据流 (Business Logic) |

### 模块二:知识运营

此模块是"人机协同"的核心,将工单变为知识。知识库由外部平台维护,此处只负责导出 csv 即可

页面分成左右两栏,左边是候选池列表,右边是清洗好的 QA 对列表

#### 3.1 候选池

**数据源逻辑**:
自动拉取满足以下条件的工单:
1. `工单状态` = **已完成**
2. `预防措施` 字段 **不为空** (有内容才有转化价值)
3. `是否已转化` = **False**

使用和"工单中心"同样的列表样式,忽略"状态"和"操作"列,然后在最开头(左边)加一个"转化"按钮,点击后打开转化弹窗

#### 3.2 转化弹窗
同样左右布局
- 左侧是该工单的详情组件,展示状态但不显示任何操作按钮
- 右侧是一个表单,上面是"问题"输入框,下面是"回答"输入框,最下面是"保存"按钮
- 左右的中间是一个右箭头按钮,文本是"AI 总结",点击后自动把工单总结成 QA 对并填写到右边
- 点击"保存"后关闭该弹窗,把该QA对作为一列增加到右侧下"清洗好的 QA 对列表",把该工单从候选池暂时移除(纯前端)

> **关于数据一致性：** 此处「保存」仅将 QA 对暂存在前端状态中，工单的 `knowledge_base` 字段**不会**立即更新。在用户点击「导出 CSV → 我已完成」之前，如果用户刷新或关闭页面，已生成的 QA 对会丢失、候选池会重新加载这些工单。这是刻意的设计——因为扣子知识库没有提供可用 API，系统无法感知外部知识库的实际入库状态，只能在用户确认后才写入数据库。Demo 阶段不实现草稿持久化（如 `qa_draft` 表），生产环境应考虑增加。

#### 3.3 清洗好的 QA 对列表

一个列表,每个条目两行,第一行是问题,第二行是回答。放不下的文字截取掉,不要自动换行。条目最右方有一个编辑按钮,可以打开3.2的转化弹窗;有一个删除按钮,能删除该行,并把这个工单放回到候选池

列表下方有一个大大的"导出csv"按钮,点击后:
1. 在前端把上述列表转化成一个表头是 "question,answer" 的 csv 文件并调用下载
2. 显示一个弹窗,告知用户前往自己的扣子知识库中上传该文件,弹窗有"我已完成"和"取消"两个按钮
3. 若点击"我已完成",则将这些 QA 对对应的工单标记为"已转化",并清空 QA 对列表,刷新候选池
4. 否则仅关闭弹窗,不改动数据和界面

# 技术栈

数据库：Supabase
前后端：将移动和PC的前后端全部用同一个 Next.js 实现，放在不同根路径，共用 api
部署：使用 Vercel 一键部署
AI 知识总结：后端配置 Kimi API key，AI 总结功能通过简单的单轮 API 调用实现（后端直接 fetch）
前端流式响应：使用 langChain useStream 前端库处理 Agent 的 SSE stream

**本次 demo 不实现的功能：** 通知推送机制
